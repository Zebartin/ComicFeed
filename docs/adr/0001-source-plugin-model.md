# ADR 0001: 源作为目录扫描的 Python 插件，无沙箱

日期: 2026-05-24

## 背景

ComicFeed 需要支持多个漫画源（exhentai, nhentai, PicaComic 等），并允许用户方便地添加新源。源需要能执行网络请求、HTML 解析、图像解码、加密解密等任意 Python 操作。

## 决策

源实现为放置在 `sources/` 目录下的独立 Python 文件（`.py`），继承 `BaseSource` 抽象类。每个源自声明身份 (`key`, `name`, `version`)、认证需求、域名列表、以及可选的依赖声明。启动时，系统扫描 `sources/` 目录，导入所有符合接口的类。

添加源的方式：
- 手动将 `.py` 文件放入 `sources/` 目录
- 通过 WebUI 上传 `.py` 文件
- 通过 WebUI 从 URL 拉取源文件

用户重启服务后新源生效。

**不引入沙箱或安全审查机制**。源代码可以执行任意系统命令。

## 替代方案

### 方案 B: JS 脚本引擎（如 venera-app）

用 JavaScript 编写源规则，通过 Python 的 JS 引擎执行。优点：天然沙箱，用户可从社区下载源文件无需担心安全。缺点：引入重型依赖（V8 绑定），调试困难，限制源的能力（如无法使用 Pillow 做图像处理）。

### 方案 C: 每源为独立 pip 包，通过 entry_points 注册

优点：标准的 Python 插件机制，依赖管理清晰。缺点：增加复杂度——写一个源需要 `setup.py`/`pyproject.toml`，不再是"扔文件就行"。

## 理由

1. **目标用户假设**：用户运行自己的 ComicFeed 实例，要么只使用内置源，要么只安装自己编写/信任的源。不存在"从不可信社区随意下载"的场景。
2. **简洁优先**：按文件即插件的模式，没有任何注册、清单、索引文件需要维护。加源 = 写文件 + 重启。
3. **安全机制可在需求出现时追加**：如果社区源生态成型，可以后续加 MD5 签名验证或 Docker 沙箱，不需要改变现有源的接口。

## 后果

- 源可以执行任意 Python 代码，用户需自行审查来源
- 源的依赖声明（`__deps__`）保持可选，常见爬虫依赖（httpx, lxml, Pillow）预装于共享 `requirements.txt`
- 源文件更新：用户手动替换文件或通过 WebUI 重新上传/拉取
