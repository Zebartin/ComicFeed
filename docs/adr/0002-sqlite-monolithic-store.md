# ADR 0002: SQLite 作为唯一数据与配置存储

日期: 2026-05-24

## 背景

ComicFeed 需要持久化多种数据：订阅配置、Gallery 元数据、下载状态、源凭证、全局设置、日志。需要选择存储方案。

## 决策

SQLite 作为唯一的持久化存储。所有数据——包括全局设置、源配置、订阅参数、Gallery 元数据、下载队列状态、系统日志——全部存入同一个 SQLite 数据库文件。WebUI 通过 ORM（SQLAlchemy）统一读写。凭证字段（cookie/token/password）使用 `cryptography.fernet` 加密后存储。

## 替代方案

### 方案 B: YAML/JSON 配置文件 + SQLite

配置用 `config.yaml`，运行时数据用 SQLite。这是更传统的布局。缺点：
- 两个数据入口，备份/迁移需要处理两处
- YAML 有格式错误风险（用户手动编辑时）
- WebUI 需要同时读写文件和数据库，代码路径分散

### 方案 C: PostgreSQL

更健壮，但 ComicFeed 是单用户单实例应用，不需要并发控制或网络访问。引入 PostgreSQL 会让简单的个人项目变成需要管理数据库服务器的复杂部署。

## 理由

1. **单文件、零配置**：SQLite 随 Python 标准库内置，部署只需要一个 `.db` 文件，备份 = 复制该文件。
2. **WebUI 统一管理**：所有配置通过 WebUI 表单修改，不存在"用户手动编辑 YAML 导致格式错误"的问题。SQLite 的 schema 即验证。
3. **足够大**：SQLite 的单文件上限为 281TB，对于漫画元数据记录绰绰有余。漫画下载量可能很大（百万级 Gallery 记录），SQLite 仍能胜任。
4. **单一迁移路径**：schema 变更只需要 ALTER TABLE 或 alembic 迁移，不需要同时维护文件和数据库的版本一致性。

## 后果

- 备份策略：定期复制 `.db` 文件即可
- 数据库文件随下载目录一起放置（或全局配置中指定路径）
- WAL 模式启用，避免读写锁冲突（调度器写 + WebUI 读）
- DB 文件大小可能随时间增长，但漫画元数据远小于图片数据
