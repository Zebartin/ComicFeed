# ADR 0003: 两阶段去重——标题相似度 + 尾部采样比对

日期: 2026-05-24

## 背景

同一漫画可能出现在多个源上（如 nhentai 和 exhentai 同时收录某同人志），也可能被同一源的多个上传者重复上传。ComicFeed 需要避免为同一内容存储多份 CBZ，同时尽量在下载完整内容之前就完成去重判定。

去重需求包括：
- **源内去重**：同一源上的重复上传
- **跨源去重**：不同源上的同一漫画
- **历史去重**：新发现的 Gallery 与已下载的历史 Gallery 比对

## 决策

采用两阶段去重策略：

### 阶段 1：标题相似度（下载前）

新发现的 Gallery 的标题经归一化处理（去除括号标签、统一大小写、去空格），与数据库中已有 Gallery 的归一化标题做字符串相似度匹配（使用 `difflib.SequenceMatcher` 或 Levenshtein 距离）。相似度超过阈值（默认 0.8）的 Gallery 归为"疑似重复"候选组。低于阈值的直接进入下载流程，不触发阶段 2。

### 阶段 2：尾页采样比对（候选组判定）

对阶段 1 标记的候选组内 Gallery，仅下载每家"最后 ≤ 10 页"（广告页最多 10 页的已知约束），鉴定广告页并计算实际有效页数。与实际页数做对比：
- 差异 ≤ 15% → 判为重复
  - 保留实际页数多者
  - 页数相同 → 保留 Gallery ID 较新者（数字 ID 比较数值，非数字 ID 比较发现时间）
- 差异 > 15% → 保留两者的 CBZ

被淘汰的 Gallery 不下载完整内容，不产生 CBZ。

历史 Gallery 已有归一化标题和实际页数存储于数据库，可直接参与比对而不需要重新下载。

## 替代方案

### 方案 A: 纯标题相似度（单阶段）

优点：完全在下载前完成，零带宽浪费。缺点：标题相似度对于漫画领域不可靠——不同漫画可能标题高度相似，同一漫画在不同源的标题格式差异可能很大导致相似度低于阈值。

### 方案 B: 基于元数据匹配

用作者名、标签集合、页数作为匹配信号，标题作为辅助。但漫画领域作者名写法有时不一致，标签由网站用户自行管理存在显著差异，页数含广告页不可靠。此方案的实际匹配效果不比标题好多少。

### 方案 C: 完整下载后比对（单阶段）

下载完整内容后再做去重。优点：准确率最高。缺点：候选组内半数带宽被浪费——当两个疑似重复的 Gallery 各 80 页时，160 页的下载只为了最终删除其中 80 页。

## 理由

1. **带宽换准确率，但折衷**：只下载尾页（≤10 页 × 候选组大小）而非完整内容，比方案 C 大幅节省带宽，比方案 A 提供更可靠的比对信号。
2. **页数是比标题更强的信号**：同一漫画的不同版本可能在标题格式上差异很大，但去广告后的实际页数通常高度接近。15% 阈值容错了少量图片损失/增加。
3. **候选组通常很小**：在"搜索条件"订阅场景下，新建画廊一次命中的数量有限（几个到十几个），候选组中的 Gallery 数更少，尾部下载的额外开销可忽略。
4. **源接口支持页码范围**：exhentai/nhentai 等源支持直接下载指定页面，阶段 2 不需下载整个 Gallery。

## 后果

- 源接口必须实现 `download_pages(gallery_id, page_range)`，支持指定页码范围下载
- 历史去重依赖数据库中存储的准确实际页数（去广告后）
- 偶尔会出现"误杀"——两本确实相似但不同的漫画被判为重复。概率低，且用户可以手动触发重下载
- 跨源去重可在订阅配置中按需开关
